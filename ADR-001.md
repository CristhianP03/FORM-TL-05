# ADR-001: Selección del proveedor de nube para el Sistema de Gestión de Soporte Técnico ISP

**Estado:** Aceptado
**Fecha:** 2026-07-21
**Autores:** Ernesto Luna Mora, Cristhian Pacheco Cárdenas, Robinson Cando Moreno

## Contexto

El sistema de gestión de tickets de soporte técnico del PFC (equipo ACC) requiere
una infraestructura que soporte un despliegue distribuido tolerante a fallos,
orquestado con Kubernetes, con replicación entre zonas geográficas de operación
(Norte, Centro, Sur). Se evaluaron tres proveedores de nube (AWS, Azure, GCP)
bajo seis criterios: costo, servicio de contenedores gestionados, bases de
datos distribuidas nativas, regiones disponibles cerca de Ecuador, SLA de
disponibilidad y facilidad de integración con Docker/Kubernetes. El detalle
completo de esta comparación se encuentra en el informe técnico del equipo
(ver `docs/matriz-decision.pdf`).

## Decisión

Se elige **Google Cloud Platform (GCP)** como proveedor de nube para alojar
el sistema.

## Justificación

- GKE (Google Kubernetes Engine) es el servicio de Kubernetes gestionado más
  maduro de los tres, al ser Google el creador original de Kubernetes, lo que
  reduce la curva de aprendizaje del equipo.
- GKE ofrece 99.95% de SLA en clústeres regionales/Autopilot, equivalente al
  mejor nivel de AWS y Azure.
- Cloud Spanner ofrece una base de datos relacional distribuida globalmente
  de forma nativa, sin necesidad de configurar manualmente la replicación
  entre zonas, lo que simplifica la arquitectura del sistema de tickets.
- GCP tiene región propia en Sudamérica (`southamerica-east1`, São Paulo),
  la más cercana geográficamente a Ecuador junto con AWS y Azure.

## Consecuencias

**Positivas:**
- Menor complejidad operativa gracias a GKE Autopilot.
- Menor curva de aprendizaje para el equipo.

**Negativas / riesgos aceptados:**
- Dependencia de un único proveedor (*vendor lock-in*), especialmente si se
  usa Cloud Spanner, que no tiene equivalente directo portable a otro proveedor.
- Persiste latencia hacia Ecuador porque ningún proveedor tiene región física
  en el país; se mitigará con CDN/caché en capas superiores si es necesario.
